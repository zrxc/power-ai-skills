# Update Rules Workflow

## 1. 目标

这份流程用来回答一个问题：一次非平凡任务做完后，哪些经验应该沉淀成长期规则，哪些只需要停留在实现里。

重点不是“每次都写文档”，而是避免高成本经验继续散落。

## 2. 适用场景

以下情况完成前，都应做一次轻量收口：

- 改了跨模块边界
- 改了 CLI 输出或治理摘要
- 改了 proposal / draft / evolution 的落盘结构
- 审核了一份会影响后续实施方向的技术方案
- 修掉了一个容易重复踩中的坑

## 3. Task Closure Protocol

完成前，用很短时间回答 4 个问题：

1. 这次有没有新 gotcha
2. 这次有没有新 task route
3. 这次有没有已有规则不够显眼
4. 这次有没有过时规则

如果 4 个问题的答案都是“没有”，可以不新增文档。

## 4. 记录阈值

只有同时满足以下 3 条中的至少 2 条，才建议把经验写进规则层：

- 可重复
- 漏掉的成本高
- 对第一次接手的人不明显

如果只是一次性的局部实现细节，不要为了记录而记录。

## 5. 写到哪里

### 稳定边界

写入：

- `.codex/rules/project-rules.md`

适合内容：

- 仓库对象语义
- 高风险边界
- 不应轻易打破的长期约束

### 任务路由

写入：

- `.codex/SKILL.md`

适合内容：

- 新增常见任务入口
- 某类任务需要先读哪些文档
- 哪些边界需要在入口处提前提醒

### 高成本经验

写入：

- `.codex/references/gotchas.md`

适合内容：

- 容易重复踩坑
- 不看实现细节就不明显
- 会直接影响治理结论或结构判断

### 用户面文档

如果经验会改变用户或维护者对命令、方案、阶段目标的理解，再同步更新：

- `docs/command-manual.md`
- `docs/technical-solutions/`
- `docs/upgrade-roadmap.md`
- `docs/upgrade-roadmap-history.md`

## 6. 写之前先检查

新增规则前先确认：

1. `.codex/` 里是否已有同类规则
2. 这次变化是新规则，还是应该把原规则前移
3. 这条内容更适合写边界、流程还是 gotcha

避免重复写出多个近义版本。

## 7. 不该做的事

- 不要把一次性的实现叙事直接抄进 gotcha
- 不要把维护者规则塞进公开 skill 文档
- 不要因为做了任务，就强行新增一条规则
- 不要把阶段路线图当成永久规则库

## 8. 收口完成的判断

当以下条件成立时，可以认为规则收口完成：

1. 新坑已经被放到合适位置，或者明确不需要记录
2. 维护者入口没有遗漏新的高频任务路径
3. 旧规则没有继续误导当前实现
