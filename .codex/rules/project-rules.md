# Project Rules

## 1. 仓库对象边界

`power-ai-skills` 是技能中心仓库，不是某个单一下游项目的本地 skill 目录。

这意味着：

- `skills/` 承载的是对外分发的公共 skill 资产
- `.codex/` 承载的是仓库维护者规则和路由
- `.power-ai/` 描述的是消费项目初始化后的真实源目录，不是当前仓库内部的维护者目录结构

不要把这三层语义混在一起。

## 2. 结构演进原则

默认优先做增量演进，不做整仓结构迁移。

只有在以下条件同时成立时，才考虑更大范围的结构重组：

- 现有目录语义已经持续误导实现或评审
- 增量补规则无法解决问题
- 重组不会混淆对外分发资产和维护者规则

如果只是想借鉴别的项目的好做法，优先迁移机制，不要先迁移目录形状。

## 3. 维护者规则放置原则

长期稳定、跨任务复用的维护者规则，应优先放在 `.codex/`：

- 稳定边界放进 `rules/`
- 收口流程放进 `workflows/`
- 高成本经验放进 `references/`

不要把这些内容直接塞进对外 skill 树，避免让公开 skill 文档承担维护者内部治理职责。

## 4. 消费项目真实源边界

当前仓库可以生成或同步消费项目的 `.power-ai/` 目录，但要始终区分：

- 仓库源码与文档
- 消费项目落盘结构
- CLI / doctor / summary 对这些结构的投影

任何涉及 `.power-ai/` 的改动，都要确认落盘结构、读取逻辑和展示逻辑是否一致。

## 5. 高风险自动化边界

以下行为默认视为高风险自动化边界：

- 自动注册
- 自动发版
- 自动 promotion
- 自动改写团队策略或治理结论

没有明确阶段目标、验收标准和回滚思路时，不要顺手扩边。

## 6. 文档方案约束

技术方案文档至少要说明：

- 适用对象
- 非目标
- 边界
- 落地步骤
- 验收标准

如果方案来自外部参考项目，还要明确：

- 哪些是直接借鉴的机制
- 哪些不适配当前仓库
- 为什么不直接照搬

## 7. 对外 skill 文档约束

公共 skill 文档是分发资产，不是维护者手册。

在修改 `skills/**/SKILL.md` 时，优先保证：

- 用途和触发条件清楚
- 切换边界清楚
- 公开文档不过度夹带维护者内部流程

## 8. 变更后的自查

如果任务改动了规则、路由、治理视图或落盘结构，完成前至少自查：

1. 是否引入了新的维护者 gotcha
2. 是否需要更新 `.codex/` 路由
3. 是否让落盘和展示再次发生漂移
