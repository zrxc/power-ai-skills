# 代码注释规则

## 目标

统一 power-ai-skills 生成代码时的注释质量，确保 AI 产物默认可读、可维护、可交接。

## 通用规则

- 默认使用中文注释
- 注释解释“为什么这样做”和“这段逻辑在防什么问题”
- 优先给复杂状态、跨组件联动、接口参数整形、边界分支补注释
- 注释要和代码保持一致，修改逻辑时同步更新注释

## 必须补注释的场景

- 搜索参数和分页参数之间存在联动
- 树节点切换会影响列表、详情或表单回填
- 弹窗、抽屉、步骤表单存在状态重置逻辑
- 表单设计器、动态配置或 schema 解析逻辑
- 请求前后有字段转换、兜底值、兼容旧接口结构
- 权限点、状态枚举、批量操作有额外限制
- 组件内部有 watcher、副作用、事件透传或复杂插槽协议

## 不建议写的低价值注释

- `给变量赋值`
- `调用接口`
- `循环数组`
- `判断是否为空`

如果注释只是复述代码字面意思，就不要写。

## 文件类型约定

### 页面文件

优先注释：

- 页面主状态的职责划分
- 搜索、分页、弹窗、树节点之间的联动关系
- 初始化加载顺序

### 组件文件

优先注释：

- props 与 emits 的契约
- 内部状态与外部状态的同步方式
- 插槽、渲染分支、兼容分支

### API 文件

优先注释：

- 接口用途
- 特殊参数来源
- 返回值是否做过二次适配

### hooks / utils 文件

优先注释：

- 输入输出契约
- 兜底行为
- 为什么要做格式转换或兼容处理

## 输出要求

- 注释尽量靠近复杂逻辑
- 一段复杂流程优先使用块注释，再用少量行内注释补充关键点
- 生成代码时，如果逻辑简单到不需要解释，可以不强行加注释
