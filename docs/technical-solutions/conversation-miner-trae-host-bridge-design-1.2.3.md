# conversation-miner trae host bridge design 1.2.3

## Goal

`1.2.3` provides the first concrete GUI host trigger sample on top of the existing auto-capture runtime.

The target tool is `Trae`.

## Problem

By `1.2.2`, the backend was ready:

- direct submit runtime
- response inbox bridge
- one-command queue-response bridge

But there was still no concrete host sample that showed how a GUI tool should hand off its confirmed final reply.

## Solution

Generate:

```text
.power-ai/adapters/trae-host-bridge.example.ps1
```

This script accepts one source:

- `-ResponsePath`
- `-ResponseText`
- `-UseClipboard`

It then:

1. normalizes the content into `.power-ai/tmp/assistant-response.trae.txt`
2. calls `trae-capture.example.ps1 -QueueResponse -ConsumeNow`
3. lets the existing bridge and runtime finish extraction, gating, and conversations write

## Why Trae First

- it was the first real pain point observed in a consumer project
- it represents the GUI host integration class well
- once this works, `cursor` / `windsurf` / `cline` / `github-copilot` can follow the same host shape

## Secondary Improvement

All generated PowerShell wrapper examples now resolve the CLI more robustly:

1. prefer local `node_modules/.bin/power-ai-skills.cmd`
2. fall back to `npx power-ai-skills`

This makes the host bridge usable in:

- installed consumer projects
- local linked development setups
- test fixtures

## Acceptance

`1.2.3` is complete when:

- `trae-host-bridge.example.ps1` is generated by `init` / `sync`
- `doctor` checks for the bridge asset
- an end-to-end test proves that the bridge script can turn a confirmed response file into `.power-ai/conversations/*.json`
