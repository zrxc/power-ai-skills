# conversation-miner cursor host bridge design 1.2.4

## Goal

`1.2.4` extends the GUI-host auto-capture sample from `Trae` to `Cursor`.

The target workflow stays the same:

1. AI decides the task is worth collecting.
2. User confirms capture.
3. The host can only provide the final confirmed response text.
4. A local bridge script turns that response into `queue-auto-capture-response --consume-now`.
5. The conversation lands in `.power-ai/conversations/`.

## Delivered Assets

`1.2.4` adds:

```text
.power-ai/adapters/cursor-host-bridge.example.ps1
```

The script accepts:

- `-ResponsePath`
- `-ResponseText`
- `-UseClipboard`
- `-KeepTempFile`

It writes a temporary response file at:

```text
.power-ai/tmp/assistant-response.cursor.txt
```

Then it calls:

```powershell
.\.power-ai\adapters\cursor-capture.example.ps1 -ResponsePath .power-ai\tmp\assistant-response.cursor.txt -QueueResponse -ConsumeNow
```

## Design Notes

- The host bridge is now generated from one shared builder instead of keeping a separate hard-coded implementation per tool.
- The bridge remains deliberately thin:
  - normalize a single response source
  - write a temp file
  - delegate capture work to the existing wrapper
- The generated wrapper examples still prefer local `node_modules/.bin/power-ai-skills.cmd` before falling back to `npx power-ai-skills`.

## Validation

`1.2.4` is complete when:

- `cursor-host-bridge.example.ps1` is generated by `init` / `sync`
- `doctor` checks the new bridge asset
- an end-to-end test proves `ResponseText -> host bridge -> queue response -> conversations`
- generated docs and release notes include the new sample
