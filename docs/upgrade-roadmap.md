# 升级路线图

本文只保留当前正在推进的一个阶段。

使用规则：

- `docs/upgrade-roadmap.md` 只保留当前活动阶段和未完成项
- 当前阶段未完成前，不在这里继续堆叠下一阶段内容
- 当前阶段全部完成后，将整段迁移到 `docs/upgrade-roadmap-history.md`
- 下一次只把新的活动阶段写回本文件，保证每次开发都能直接看到“现在要做什么”

历史记录：

- 已完成阶段、已收口方案、历史版本规划统一迁移到 `docs/upgrade-roadmap-history.md`

## 当前阶段

### P5-6 项目评估与结构优化第一版

阶段目标：

- 基于当前仓库现状完成一轮工程评估收敛，把“可以继续做功能”先建立在“结构可维护、发布可控、校验稳定”的基础上
- 优先解决已经识别出的结构性问题：命令注册重复维护、发布包边界偏宽、热点模块持续膨胀、仓库中存在低价值占位与冗余产物
- 在不改变现有治理边界的前提下完成安全清理、结构整理和验证，为后续功能扩展留出更稳定的工程底座

本阶段只做：

- 梳理 CLI 命令注册、命令路由与 project root 解析链路，收敛重复维护的命令元数据
- 评估并收缩 npm 发布清单，明确哪些 `docs`、`manifest`、运行时产物属于必须随包发布，哪些只保留在仓库或发布流程中
- 对已确认的冗余内容做安全清理，例如空目录、未使用占位、无价值中间产物，同时保持现有命令与测试行为不回归
- 对热点模块做第一轮结构拆分或边界约束，优先控制超大入口文件和多职责模块继续膨胀
- 把本轮评估结论同步到路线图、命令文档和必要的维护文档中
- 保持 `pnpm test`、`check-package`、关键 CLI 入口和发布检查可通过

本阶段不做：

- 新增运营趋势视图、治理趋势聚合或新的报表能力
- shared skill 自动正式晋升
- wrapper 自动注册
- release 自动发版
- project-local 自动晋升到 `manual` 或 shared skill
- 跨仓库或远端遥测采集
- 大规模重写现有业务能力或替换当前治理模型

## 完成标准

- 仓库已经有一份可执行的项目优化结论，明确说明当前问题、已做收敛和下一步建议，而不是只停留在口头判断
- 新增或调整命令时，不再需要在多个位置重复维护同一批命令元数据
- npm 发布包边界收敛到可解释范围，运行必需资产、文档资产和中间产物的职责清晰
- 已确认的冗余内容完成安全清理，且 `pnpm test`、`check-package`、关键 CLI 命令验证通过
- 至少一个热点模块的职责边界得到收敛，后续继续拆分时有明确落点，不再继续向单文件堆积
- 本阶段文档与实现一致，后续进入新阶段时可以直接基于当前收口结果继续推进
